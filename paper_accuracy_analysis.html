Detailed Analysis by Section

1. Backend Technology Stack ❌

Paper claims: "The backend is Flask-based with key files ai_controller.py, static_scan.py, and dao_interface.py."

Reality: The backend is built with Next.js / TypeScript. The equivalent files are:

  • scanner.ts (not static_scan.py)
  • AgentSystem.ts (not ai_controller.py)
  • ReportRegistry.sol / smart contract interactions (no dao_interface.py)

There is no Python or Flask anywhere in the project.

2. Static Analysis Tools ❌

Paper claims: "Integrates Slither and Mythril (static analysis tools)."

Reality: The project uses 10 different tools running in parallel:

Semgrep, PatternScanner                      (universal / multi-language)
NpmAudit, RetireJS, Njsscan                  (JavaScript / TypeScript)
Bandit, PipAudit                             (Python)
Gosec                                        (Go)
SpotBugs                                     (Java / Kotlin)
PHPStan                                      (PHP)
Brakeman                                     (Ruby / Rails)

Slither and Mythril are Solidity-specific tools. The project instead scans general web application repositories (GitHub URLs or live websites). The paper describes smart-contract analysis tools while the code actually performs web/app security scanning.

3. Smart Contracts & DAO Mechanism ✅

This is the most accurate section of the paper. The contracts directly match:

  • BountyVault.sol — lockReward() / releaseReward() with a 5% platform fee. Correct.
  • ReportRegistry.sol — Validators stake 0.01 ETH, votes tallied (2-of-3 quorum), stakes slashed for incorrect votes. This is a real DAO governance mechanism.
  • ValidatorReputation.sol — On-chain reputation; canValidate() gates participation. Correct.
  • IPFS hash stored in submitReport(). Correct.

The paper's description of "decentralized triage" and "automated bounty payout" is a fair and accurate description of the contract architecture.

4. Multi-Agent / MCP Architecture ✅ (Conceptually)

Paper claims: "MCP-inspired architecture coordinating Static Analysis Agent, AI Semantic Agent, Dynamic Validation Agent."

Reality: AgentSystem.ts exports distinct agent functions:

  • validateTarget() — Pre-scan validation agent
  • performRecon() / performRepoRecon() — Reconnaissance agent
  • reasonVulnerabilities() / reasonRepoVulnerabilities() — LLM reasoning agent
  • generateSecurityReport() — Report generation agent

The pipeline in scanner.ts runs static tools in parallel then feeds results into LLM reasoning — which does match the paper's description. The "MCP-inspired" framing is plausible given the modular agent structure.

5. Blockchain Network ❌

Paper claims: "Demonstrated on the Polygon testnet."

Reality: hardhat.config.ts only configures localhost (http://127.0.0.1:8545). There is no Polygon, Mumbai, Amoy, or any other public testnet configured. The deployment was local, not on Polygon.

6. Confidence Score Formula ❌

Paper claims: A weighted confidence score C = (0.4·S) + (0.35·A) + (0.25·D) with a 95% threshold.

Reality: No such formula exists in the codebase. generateSecurityReport() hardcodes confidence: 1.0 for every finding. The actual confidence is purely the LLM's output classification (severity: LOW/MEDIUM/HIGH/CRITICAL) without any mathematical scoring model.

7. Experimental Results ❌ (Unverifiable)

Paper claims: 91.9% accuracy, 45-second triage time, 68% false-positive reduction vs SmartBugs/SolidiFi benchmarks.

Reality:

  • The project scans live GitHub repos and web URLs — it does not have a benchmark dataset pipeline.
  • There are no test result files, benchmark scripts, or evaluation logs that would produce these numbers.
  • These figures appear to be fabricated or from a theoretical/aspirational description not grounded in the actual code.

8. Database ✅

Reality: schema.prisma uses PostgreSQL with models for User, Session, Scan, Finding, and Consent. The persistence layer is production-grade and consistent with what the paper implies.
